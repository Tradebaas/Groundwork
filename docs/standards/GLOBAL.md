# GLOBAL: universal code standards (stack-independent)

These hold in every language and stack. Stack-specific standards live next to this file,
generated by the `stack` skill; where they conflict, the stack file wins and must say why.

## Code

- Follow the decision ladder (AGENTS.md) before writing anything: need → reuse → platform →
  minimum that works. No speculative abstractions, no options nobody asked for.
- Names say what things are; functions do one thing; modules own one concern. If a senior in
  this stack would raise an eyebrow, it's wrong: idiom over cleverness.
- Source files stay under the `codeFileMaxLines` budget in `checks/config.json` (500 by
  default, enforced by the `code-file-cap` check); aim well under it, around 300. A function
  that no longer fits on one screen is doing two jobs: split it.
- Deep modules: expose few, small, stable entry points that hide real implementation depth
  behind them. A module whose interface is as large as its implementation adds cost without
  hiding anything. No pass-through layers that only forward to the next layer.
- Comments state what the code cannot: constraints, invariants, the *why* of the non-obvious.
  Never narrate the next line, never leave commented-out code. Git remembers; delete it
  (the zombie-code check catches blocks of it, but the rule is: none at all).
- Errors: handle at trust boundaries and where data loss threatens; never swallow silently;
  fail with a message that names the cause. The full contract is the errors & observability
  floor below; user-facing errors follow `docs/design/VOICE.md`.
- No new dependency when the platform or an existing dependency does the job. Every new
  dependency is a decision: license checked, maintenance checked, recorded in the spec.

## Tests

- Non-trivial logic leaves at least one runnable check behind, always.
- Red before green at the testing seams agreed in the spec: write the failing test at the
  seam first, then make it pass. Refactoring belongs to the review stage, never mixed into
  the implementing diff.
- Test behavior, not implementation. See each new test fail for the right reason once: break the
  code or the input on purpose and read the failure message. A test that cannot fail (asserting on
  a value the test itself supplied, assertion never reached, mock asserting itself) is worse than
  no test: it reports safety that is not there.
- Bug fix = regression test first, then the fix.

## Git

- Conventional Commits v1.0.0 (`feat:`, `fix:`, `docs:`, `chore:`, ...), imperative, scoped small.
- Every commit ends with a `Traces-to:` trailer naming the scope item it serves (`Traces-to: SC-3`)
  or the request it answers (`Traces-to: explicit request: <what was asked>`). Gated by the
  `commit-msg` hook and re-proven in CI. This is what lets a sha resolve back to a requirement,
  and what makes the change list a system can generate instead of a person remembering.
- Land PRs by rebase, not squash: squash replaces your message with one the host composes, which
  wraps long values and appends its own trailer block, leaving the trace unreadable to git
  (decision 0014). Turn squash off at the host rather than relying on remembering.
- Never commit directly to `main` once CI exists; never force-push shared branches;
  never `--no-verify`. Secrets never enter history, not even for a minute.

## Security (floor, all stacks)

- Validate all input at trust boundaries; encode all output for its destination.
- Least privilege everywhere: tokens, DB users, file permissions, CI scopes.
- Secrets in the environment or a vault, `.env.example` documents the shape.
- Dependencies audited in CI; known-vulnerable versions block delivery.

## Errors & observability (floor, all stacks)

- Two kinds of errors, two responses. Operational errors (timeout, bad input, a dependency
  down) are expected: handle them with a fallback, a bounded retry, or a clear user message.
  Programmer errors (broken invariant, impossible state) mean the state is untrusted: fail
  fast and loud, never continue in a corrupted state. The stack file picks the idiom
  (crash-and-restart under a supervisor for servers; error boundary plus report for UI).
- Every caught error is handled, given context, or rethrown; one global handler is the
  safety net, not the strategy. Multi-step operations fail closed: roll back what was done
  rather than leave half a transaction (OWASP A10:2025).
- Users get a short, helpful message per `docs/design/VOICE.md`; diagnostics (stack traces,
  internals, versions) go to the logs only, never to the user.
- Every remote call has a timeout; retry only transient failures, bounded, with backoff and
  jitter; a circuit breaker guards integration points that fail repeatedly. The stance per
  integration is set in the architecture's contracts.
- Logs are structured, use their levels consistently, and carry a correlation ID from the
  request's entry point so one failure can be followed across the system. Secrets, tokens
  and sensitive personal data stay out (OWASP A09:2025; logs are a GDPR surface).
  Security-relevant events (auth, denied access, validation failures) are logged.
- The critical flow exposes a health signal, and alerting fires on symptoms users feel, not
  on internal causes (SRE). This floor is built during construction; the `maintain` skill
  verifies it exists at launch, it does not retrofit it.

## Accessibility (floor, all user interfaces)

- WCAG 2.1 AA is the legal baseline in the EU (see `docs/compliance/COMPLIANCE.md`): semantic
  structure, keyboard operability, contrast, focus visibility, labels. Not a polish step; built in.
